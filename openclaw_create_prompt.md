# OpenClaw 一键安装部署 Prompt（Docker 优先 / 小白黑盒版）

你现在是一名资深 Linux / DevOps / Python 部署工程师。请为 OpenClaw 设计并输出一套“面向小白用户的黑盒式一键安装部署方案”。

核心定位：
- 对用户：开箱即用。用户不需要理解 Docker、Compose、Python 虚拟环境、系统依赖、镜像构建、容器网络等概念，只需要运行脚本并按提示选择。
- 对实现：工程化落地。优先使用 Docker / Docker Compose 封装运行环境，把复杂度收敛到安装脚本、Compose 编排、镜像构建、升级回滚和卸载逻辑中。
- 对结果：必须能落地。不要给思路稿、伪代码或半成品脚本，必须输出完整方案和完整代码。

硬性前提：
1. 当前阶段只输出代码和说明，不要真的执行安装、拉镜像、改系统配置。
2. 必须先分析 OpenClaw 仓库中的真实文件，再决定部署方式；禁止凭空假设启动命令、端口、配置文件路径、健康检查方式、镜像结构。
3. 如果仓库信息不足，请明确列出缺失项、保守默认值和 TODO，不允许用臆测填空。
4. 默认采用 Docker / Docker Compose 方案；只有在仓库事实明确表明容器化不可行时，才允许退回原生安装，并必须说明理由。

## 一、任务目标

请输出一套完整、可落地、对小白友好的 OpenClaw 一键安装方案，至少包含以下文件：

1. `install_openclaw.sh`
2. `upgrade_openclaw.sh`
3. `uninstall_openclaw.sh`
4. `compose.yaml` 或 `docker-compose.yml`
5. `Dockerfile`（如果项目需要自建镜像）
6. `.env.example`
7. 配套使用说明文档

总体要求：
- 用户只需要运行一个脚本并按提示选择，即可完成安装、升级、卸载、状态检查等操作。
- 内部自动完成环境检测、Docker/Compose 安装、镜像拉取或构建、容器启动、开机自启、健康检查、日志定位、故障提示。
- 尽量兼容主流 Linux 发行版：Ubuntu / Debian / CentOS / RHEL / Arch / Kali。
- 方案必须高容错、可重复执行、可回滚、可维护。

## 三、方案设计原则

### 1. 用户黑盒化

用户看到的只能是简单动作，例如：
1. 标准安装
2. 国内网络优化安装
3. 升级 OpenClaw
4. 卸载 OpenClaw
5. 查看运行状态和日志

不要要求用户：
- 手动输入复杂 Docker 命令。
- 手动理解镜像、卷、网络、Compose、Dockerfile。
- 手动修复大部分可自动化的问题。

### 2. Docker 优先但不向用户暴露 Docker 复杂度

默认通过 Docker / Docker Compose 部署，理由包括：
- 统一运行环境，减少跨发行版差异。
- 避免污染宿主机 Python 和系统依赖。
- 更容易实现升级、回滚、重装、迁移和卸载。
- 更适合小白“脚本一键拉起”的使用方式。

### 3. 向企业级可维护性靠拢

方案必须吸收以下工程特性：
- 安装前做完整预检，而不是边装边碰运气。
- 每一步有结构化日志和明确的成功/失败状态。
- 支持重复执行，不会越装越乱。
- 支持升级、回滚、备份、卸载。
- 对宿主机改动有边界、有确认、有备份。

### 4. 小白友好

必须做到：
- 欢迎信息清楚说明“脚本会自动完成什么”。
- 提示文案尽量使用自然语言，不滥用专业术语。
- 高风险操作必须确认。
- 出错时明确告诉用户：为什么失败、日志在哪、下一步怎么做。
- 安装完成后输出一段清晰 summary，包括启动方式、访问方式、日志查看方式、升级方式、卸载方式。

### 5. 安全保守

禁止：
- 静默删除已有目录。
- 静默覆盖已有配置。
- 静默修改 Docker daemon 配置。
- 静默删除用户数据。
- 默认卸载系统级共享依赖。

## 四、必须实现的能力

### A. 环境检测与资源预检

安装脚本必须自动检测：
- Linux 发行版和版本，优先通过 `/etc/os-release` 识别。
- 包管理器：`apt` / `yum` / `dnf` / `pacman`。
- CPU 架构：至少识别 `x86_64`、`aarch64`。
- 当前用户是否为 root，若不是则检测 sudo 能力，必要时提示提权。
- Docker 是否已安装。
- Docker Compose 是否可用，优先支持 `docker compose`，必要时兼容 `docker-compose`。
- systemd 是否可用。
- HTTPS 网络连通性，优先使用 `curl` / `wget`，不要只依赖 `ping`。

还必须做资源预检：
- 检查可用内存。
- 检查磁盘剩余空间。
- 检查安装目录所在分区是否可写。
- 检查需要暴露的端口是否被占用。
- 检查系统是否已配置 SWAP，以及 SWAP 容量是否足以覆盖低内存机器的基础缓冲。

资源与端口预检要求：
- 如果仓库文档中存在明确要求，优先采用项目官方要求。
- 如果仓库未说明，则使用保守默认阈值，并在 `Missing Information` 中写明这些阈值来自保守假设，不是项目事实。
- 如果检测到机器内存偏低（例如低于保守阈值 4GB，除非仓库有更明确要求），且系统未配置 SWAP 或 SWAP 过小，脚本必须明确告警。
- 在上述低内存场景下，脚本可以提供“辅助创建 swapfile”的可选能力，例如 2GB SWAP，但必须满足以下条件：
  - 只能作为用户显式确认后的辅助功能，不能默认开启。
  - 必须说明会修改哪些系统配置、占用多少磁盘、如何撤销。
  - 若用户拒绝创建 SWAP，脚本仍应继续，但需要明确提示后续可能出现 OOM、构建失败或容器被系统杀死的风险。
- 在低内存场景下，如仓库允许，优先选择拉取预构建镜像而不是本地 `docker build`，并尽量降低构建阶段的并发和缓存开销。

### B. Docker 安装与适配

脚本必须支持：
- Docker 未安装时自动安装。
- Docker 已安装时检查版本兼容性。
- Compose 插件优先，旧版 `docker-compose` 兼容作为次选。
- Ubuntu / Debian、CentOS / RHEL、Arch / Kali 的安装差异处理。
- 旧系统或不受支持系统的明确告警。

网络与镜像要求：
- 在网络受限时，给出国内镜像、代理或离线镜像导入方案。
- 如果需要修改 Docker registry mirror，必须先备份原配置并征求确认。
- 不得无提示地改写用户现有 Docker daemon 配置。
- 如果需要基于源码执行 `docker build`，必须在 Dockerfile 和构建流程中支持通过 `ARG` 或等价机制注入以下参数：
  - `HTTP_PROXY`
  - `HTTPS_PROXY`
  - `NO_PROXY`
  - 与包管理相关的镜像源参数，如 `PIP_INDEX_URL`、`PIP_EXTRA_INDEX_URL`、`APT_MIRROR`
- 若检测到用户位于国内网络环境，或用户显式指定 `--mirror` / `--proxy`，脚本必须能够将上述参数动态透传到 `docker build` 过程，而不是只配置宿主机 Docker 拉镜像。
- 如果镜像构建阶段涉及 `apt update`、`pip install`、模型下载、Git 拉取等联网步骤，必须为这些步骤提供国内网络可用的替代策略，例如：
  - 动态切换 apt 源、pip 源
  - 使用代理
  - 使用预下载缓存
  - 使用镜像站或可配置下载端点
- 如果项目依赖 HuggingFace 或类似模型仓库，必须支持通过环境变量、代理或镜像端点配置下载路径，并优先考虑缓存目录挂载，避免重复下载导致超时或卡死。
- 不要把特定国内源硬编码为唯一默认值；应优先做成可配置项，并在交互模式下给出推荐选项。

### C. OpenClaw 部署逻辑

必须做到：
- 基于仓库真实内容识别 OpenClaw 的启动方式。
- 如仓库已有镜像，优先评估是否直接使用。
- 如仓库无可用镜像，则基于源码构建 Docker 镜像。
- 使用 Compose 管理服务、端口、卷、环境变量、重启策略。
- 自动准备安装目录、数据目录、配置目录、日志目录。
- 自动生成 `.env` 或同类配置文件。
- 自动执行拉镜像或构建镜像。
- 自动启动容器。
- 自动执行启动验证或健康检查。
- 如项目存在模型缓存、下载缓存或构建缓存目录，优先使用持久化卷或宿主目录挂载，避免重复下载。

部署验证要求：
- 验证必须基于项目真实入口。
- 如果是 Web / API 服务，优先使用真实 HTTP 健康检查接口。
- 如果是 CLI 或后台任务，使用真实入口命令、进程状态或日志信号校验。
- 禁止编造如 `openclaw --version` 之类未经证实的命令。

容器日志要求：
- Compose 中的每个长期运行服务都必须定义明确的日志策略，防止容器日志无限膨胀占满磁盘。
- 如果使用 Docker 默认 `json-file` 日志驱动，必须设置大小和轮转上限，例如 `max-size`、`max-file`。
- 如果使用其他日志驱动，也必须提供等价的限额或宿主机侧日志轮转方案，并在说明文档中解释。
- 日志策略必须与“安装日志文件”区分开：前者针对容器运行日志，后者针对安装脚本执行日志。

### D. 开机自启与宿主管理层接管

必须考虑服务接管，而不是只做一次性 `docker compose up -d`。

要求：
- Compose 服务本身应配置合理的重启策略，如 `unless-stopped`。
- 若系统支持 systemd，应优先提供一个“管理 Compose 栈”的 systemd unit，而不是假装直接用 systemd 管 OpenClaw 进程本体。
- 如果选择生成 systemd unit，必须说明 unit 的职责、依赖关系、日志行为和启停命令。
- 如果不生成 systemd unit，必须说明为什么仅依赖 Docker 自启策略即可。

### E. 容错、重试与回滚

脚本必须：
- 使用 `set -euo pipefail`。
- 使用 `trap ERR INT TERM EXIT`。
- 记录当前步骤、失败命令、返回码、日志位置。
- 对下载、安装 Docker、拉镜像、构建镜像等关键操作自动重试，默认最多 3 次，可带退避策略。
- 部署失败时执行安全回滚。

回滚要求：
- 默认只回滚脚本自身创建的内容。
- 包括当前脚本创建的目录、临时文件、容器、Compose 临时文件、网络、卷、systemd unit、软链接等。
- 不得默认删除用户已有 Docker 配置。
- 不得默认删除共享系统依赖。
- 不得默认删除业务数据，除非用户在卸载或重装流程中明确确认。

### F. 幂等性与重复执行

脚本必须支持重复执行，且行为可预测：
- 检测已有安装。
- 检测已有目录是否确实属于 OpenClaw。
- 避免重复创建容器、卷、网络、配置文件、systemd unit。
- 已安装时，提供清晰选项：
  1. 覆盖重装
  2. 保留现有版本并修复环境
  3. 执行升级
  4. 退出
- 非交互模式下允许通过参数指定上述行为。

### G. 参数化与配置管理

所有关键配置必须集中管理，且支持命令行参数覆盖，例如：
- `--dir`
- `--version`
- `--channel`
- `--mirror`
- `--proxy`
- `--port`
- `--non-interactive`
- `--force`
- `--skip-docker-install`
- `--skip-healthcheck`
- `--dry-run`

配置设计要求：
- 顶部集中定义默认变量。
- 生成的 `.env`、Compose 文件、脚本默认值要彼此一致。
- 安装、升级、卸载三个脚本之间的变量命名、目录结构、服务名必须统一。
- 如果仓库中存在必填敏感配置，例如 API Key、Access Token、数据库密码、模型服务密钥等，脚本必须先基于仓库事实识别这些字段，而不是凭空捏造。
- 在交互模式下，若检测到这些必填敏感变量，脚本应主动以安全方式提示用户输入，并写入 `.env` 或等价配置文件。
- 输入敏感变量时应尽量避免明文回显，例如使用 `read -s` 或等价安全输入方式。
- 若用户暂时跳过敏感变量输入，脚本必须给出清晰警告，说明哪些功能将不可用，以及后续应去哪个文件补填。
- 生成的敏感配置文件应采用最小必要权限，例如 `.env` 写入后设置为仅当前用户可读写的权限。
- 不得把敏感变量直接硬编码进 Compose 文件、安装脚本正文或会进入 shell history 的命令行参数中，除非仓库已有明确约定且不存在更安全做法。

### H. 日志、诊断与交付体验

脚本必须：
- 在终端输出格式化日志，如 `[INFO]` / `[WARN]` / `[ERROR]`，必要时使用颜色高亮。
- 同时将完整日志写入文件。
- 日志优先写入 `/var/log/openclaw_install.log`。
- 若无权限则回退到 `$HOME/.local/state/openclaw/install.log`。
- 若仍不适用，则回退到安装目录下的日志文件。

日志至少包含：
- 时间
- 级别
- 当前步骤
- 执行命令
- 返回状态
- 失败原因

安装完成后必须输出可直接复制使用的运维摘要，包括：
- 查看服务状态的命令
- 查看容器日志的命令
- 重启命令
- 升级命令
- 卸载命令
- 配置文件位置
- 数据目录位置
- 访问地址或服务入口

### I. 升级与卸载

请同时设计升级脚本和卸载脚本。

升级脚本必须：
- 支持拉取新镜像或重建镜像。
- 升级前备份关键配置。
- 升级失败时支持回滚到旧版本。
- 升级后自动做健康检查。

卸载脚本必须至少提供三种模式：
1. 仅停止并删除容器
2. 删除容器和镜像
3. 删除容器、镜像、卷、配置、数据

卸载要求：
- 删除数据前必须二次确认。
- 必须告知会删除什么、会保留什么。
- 卸载完成后输出残留路径说明。

## 六、硬性禁止事项

以下行为禁止出现：

1. 禁止假设不存在的 OpenClaw 命令、固定端口、固定配置文件路径，除非仓库中有明确证据。
2. 禁止把复杂操作大面积甩给用户手动完成，除非确实无法自动化。
3. 禁止默认执行高风险操作，例如静默删目录、静默覆盖配置、静默修改 Docker daemon 配置、静默删除用户数据。
4. 禁止把原生宿主机安装作为主方案，除非容器化被仓库事实明确否定。
5. 禁止输出只有框架没有内容的伪代码。
6. 禁止让安装、升级、卸载三个脚本在路径、服务名、数据目录、日志目录上互相冲突。

## 七、输出格式要求

请严格按以下顺序输出最终答案：

### 1. `Repository Findings`

- 关键文件和判断依据

### 2. `Deployment Decision`
- 为什么采用当前部署方式
- 是否使用现有镜像或自建镜像
- 健康检查依据
- 是否采用 systemd 接管 Compose

### 3. `Missing Information`

- TODO 项

- 如果没有缺失，也要明确写“无关键缺失信息”
