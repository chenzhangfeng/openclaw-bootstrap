# Novice Release Delivery Planning

## Goal

- 目标：把 `docs/novice-release-checklist.md` 落成可执行交付线，优先完成面向小白用户的 `Windows fat` 正式包闭环，再推进 macOS。
- 范围：
  - 本仓库可控的打包、启动器、说明文档、用户入口文案、构建校验与发行约束。
  - 以“先启动，再在产品页内配置 API；脚本配置仅作兼容兜底”为正式方向。
  - 先修正式包生成链路，再收敛终端用户入口。
- 非目标：
  - 当前回合不直接修改 OpenClaw 产品页面内部实现。
  - 当前回合不设计 Git 更新以外的完整自动升级系统。
  - 当前回合不承诺完成 macOS `.app` / `.dmg` 成品，只先为其留出阶段位。

## Context

- 涉及模块：
  - `build/build-windows.ps1`
  - `README.md`
  - `launchers/windows/*`
  - `openclaw-portable/*`
  - `portable/README.md`
- 关键约束：
  - 仓库当前未包含 `openclaw/` 应用源码，应用内 API onboarding 只能先作为产品契约与接口方向记录，不能在本仓库内直接完结。
  - 当前工作区存在其他文档删除/变更，需采用窄提交，避免误碰无关改动。
  - 小白正式发行物必须优先保证“拿到包即可启动”，不能依赖 Git、终端、`chmod`、`pnpm` 概念暴露给用户。
- 现有实现摘要：
  - Windows 启动链路已具备便携 Node / pnpm 思路，但构建脚本存在参数解析问题，且对正式包必需资产缺少明确校验。
  - 用户文案仍把“先配置 API Key，再启动”作为推荐流程，与目标方向不一致。

## Architecture Notes

- 主要改动点：
  - 项目级分阶段顺序固定为：
    1. 小白成品契约与执行文档落地
    2. Windows fat 构建链路修复与正式包约束
    3. Windows 用户入口文案与兼容脚本收敛
    4. macOS 小白分发形态设计
    5. 验证矩阵与发版门槛回补
  - 兼容策略：
    - 保留现有 API 配置脚本作为临时兼容兜底。
    - 正式入口改为“先启动进页面”。
- API 契约影响：
  - 本仓库侧只调整入口契约与配置路径说明，不新增新的 API 数据格式。
- 数据库迁移影响：
  - 无。
- 安全影响：
  - 入口文案去前置化后，减少用户在脚本层直接接触密钥的频次。

## Risks

- 风险：
  - OpenClaw 本体页面内若尚无成熟的模型配置入口，则“先启动后配置”只能先以目标契约和兼容兜底并行存在。
  - 缺少 `openclaw/` 源码与预置 Playwright 浏览器资产时，无法在当前仓库内完成真正的 fat 包全链路 smoke。
- 回滚方式：
  - 本阶段修改集中于构建脚本和文案，可逐文件回退。
