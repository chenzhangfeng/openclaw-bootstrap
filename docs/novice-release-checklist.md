# OpenClaw 小白用户发行清单（Windows / macOS）

## 1. 产品目标

- 目标用户：电脑基础较弱、希望少折腾的个人用户。
- 目标平台：`Windows` 为主，`macOS` 次之。
- 目标体验：用户拿到包后，只需要解压、双击、等待浏览器打开，不需要先学会命令行、Git、Node、pnpm、Docker。
- API 策略：`不要把 API Key 配置做成启动前置步骤`。应允许 OpenClaw 先启动，再在产品页面里集中完成：
  - 选择模型供应商
  - 查看购买入口
  - 查看填写说明
  - 填写 API Key
  - 或选择先跳过 / 使用本地 Ollama

## 2. 成品定义

面向小白用户的正式发行物，建议只保留两类：

- `Windows fat 包`
  - 解压即用
  - 不依赖用户本机已有 Node / Git / pnpm
  - 首次启动不要求联网下载核心依赖
- `macOS fat 包`
  - 最终形态优先是 `.app` / `.dmg`
  - 不要求用户执行 `chmod +x`、`./start.sh`
  - 首次启动不要求终端操作

以下形态不应作为“小白正式发行物”：

- `slim 包`
- Git 仓库分发
- 需要终端输入命令的脚本包
- 需要用户预先填写 API Key 才能启动的包

## 3. P0 发布阻断清单

以下项目未完成前，不建议对小白用户公开分发：

- [ ] 首次启动路径收敛为单一路径：`解压/打开 -> 双击启动 -> 浏览器打开 -> 页面内引导`
- [ ] 删除“先运行 API 配置脚本再启动”的产品前置要求
- [ ] Windows 正式包预置完整运行时：`node`、`pnpm`、`openclaw`、`node_modules`、`Playwright browsers`
- [ ] macOS 正式包改为 Finder 可直接启动的形态，而不是 `.sh` 手工执行
- [ ] 正式包不再依赖 Git 更新，不再要求用户理解分支、stash、reset
- [ ] 构建脚本直接产出面向终端用户的成品包，而不是半成品目录
- [ ] 首次启动失败时，给出用户可执行的友好提示，而不是一串工程化报错
- [ ] 明确并验证“无 API Key 也能先进入产品页”的启动链路

## 4. P1 产品与工程清单

### 4.1 首启体验

- [ ] 启动后自动打开浏览器
- [ ] 若端口已被本机已有实例占用，优先提示“已检测到正在运行的 OpenClaw”，并直接打开已运行地址
- [ ] 若首次初始化较慢，显示明确进度提示，不让用户误以为卡死
- [ ] 若缺少关键运行依赖，提示“如何修复”，而不是只提示“启动失败”

### 4.2 API 配置内置化

- [ ] 首页或首次引导页内提供“配置模型”入口
- [ ] 页面内列出常用供应商：`OpenAI`、`DeepSeek`、`通义`、`Kimi`、`Yi`、`Ollama`、自定义兼容接口
- [ ] 页面内提供购买入口 / 官方链接 / 计费说明入口
- [ ] 页面内提供 base URL 和 API Key 的填写说明
- [ ] 页面内支持“先跳过，稍后再配”
- [ ] 页面内支持“Ollama 本地模式”
- [ ] 页面内保存后的结果可立即用于后续对话，不要求重启整个分发包

### 4.3 Windows 用户体验

- [ ] 尽量兼容常见路径：桌面、下载目录、中文用户名目录
- [ ] 若无法完全兼容，考虑首启自动搬运运行时到 `%LOCALAPPDATA%` 等受控目录
- [ ] 对杀毒软件拦截给出明确提示文案
- [ ] 对缺少 VC++ 运行库给出显式修复入口
- [ ] 不要求用户手工安装 Node / Git / Python

### 4.4 macOS 用户体验

- [ ] 处理 Gatekeeper / 未验证开发者提示
- [ ] 避免要求用户打开终端执行 `chmod`
- [ ] Apple Silicon 与 Intel 分开发包或做清晰区分
- [ ] 页面或 README 中写清首次打开的系统提示处理方式

### 4.5 更新策略

- [ ] 小白包默认不暴露 Git 更新入口
- [ ] 更新策略改为“下载新版本覆盖”或“应用内检查新版本并跳转下载页”
- [ ] 若保留“检查更新”按钮，文案必须是产品语言，而不是 Git 语言
- [ ] 更新后保留用户已有配置与数据

### 4.6 构建与发布

- [ ] Windows 构建脚本修复参数问题，保证可稳定产出 `fat` 成品
- [ ] macOS 构建链路补齐面向终端用户的打包步骤
- [ ] 发布流水线自动收集 Playwright 浏览器内核
- [ ] 发布流水线自动验证成品目录完整性
- [ ] 发布产物命名区分平台、架构、版本、渠道

## 5. 验证清单

正式分发前，至少完成以下 smoke test：

- [ ] 纯净 Windows 10 x64 机器验证
- [ ] 纯净 Windows 11 x64 机器验证
- [ ] 纯净 macOS Apple Silicon 机器验证
- [ ] 纯净 macOS Intel 机器验证
- [ ] 机器上没有预装 Node / Git / pnpm 时验证
- [ ] 中文用户名 / 桌面路径 / 下载目录路径验证
- [ ] 首次离线启动 `fat` 包验证
- [ ] 未配置 API Key 时可进入产品页验证
- [ ] 页面内配置 API 后可立即正常对话验证
- [ ] Ollama 模式链路验证
- [ ] 更新后配置保留验证
- [ ] 日志与报错文案可被非技术用户理解验证

## 6. 当前仓库与目标的主要差距

基于当前代码现状，和“小白分发目标”之间的主要差距如下：

- 目前 `README.md` 仍然把“先配置 API Key，再启动”作为推荐流程
- Windows / macOS 启动器仍然保留脚本式 API 配置入口
- macOS 当前仍是 shell 脚本启动流，不是 App 分发流
- 更新逻辑仍基于 Git 仓库，而不是面向分发包的更新模型
- 构建脚本目前只创建 `browsers/` 目录，没有把浏览器内核自动打进成品
- Windows 路径策略仍偏保守，对小白常见目录不够友好
- Windows 构建脚本本身还需要修正，才能稳定产出正式包

## 7. 建议执行顺序

建议按下面顺序推进，而不是同时铺太多线：

1. 先定义统一的“小白成品契约”
   - 用户只做解压和双击
   - API 在页面内配置
   - 不暴露 Git / shell / chmod / reset / pnpm 概念
2. 先做 Windows fat 正式包闭环
   - 因为这是主目标平台，且验证成本最低
3. 再做 macOS 正式包
   - 目标直接对齐 `.app` / `.dmg`
4. 最后再考虑更新与自动升级
   - 不要让更新机制阻塞首发体验

## 8. 完成标准

只有当下面 4 条同时满足时，才算真正达成“小白可用”：

- 用户拿到包后，不需要查教程就能完成首次启动
- 用户不需要预先理解 API Key、base URL、pnpm、Git、终端
- 用户即使暂时不填 API，也能先进入产品页面
- 用户在页面内能清楚看到“去哪里买 API、怎么填、填完后怎么开始用”
